# 内容确认协议（L2→L3）

> 路径：references/confirmation/editorial-confirmation.md
> 职责：定义 L2（内容资产包）→ L3（格式消费）的内容确认关卡。正文结构、事实准确性、引用完整性、图片资源。
> Phase：1
> 依赖：CONSTITUTION.md, schemas/content-package.md
> 被依赖：workflows/*.md
> 最后更新：2026-06-17

---

## 一、确认时机

L2 content-package（manifest.json + body.md）生成完成之后、L3 格式消费之前。正文和素材都已备好，需确认内容质量后再进入格式化。

---

## 二、确认四项

### 2.1 正文结构是否合理

**确认内容：**

- body.md 的章节结构是否符合该类型的内容结构指南（templates/content-structure/<type>.md）？
- 叙事线是否清晰：是否有逻辑断裂/跳跃？
- 篇幅是否在合理范围内（不偏长不偏短）？
- 视觉引导块的位置是否合理（"此处需要配图/对比表"等标注）

**呈现方式：**

```
📝 正文结构审查
━━━━━━━━━━━━━━━━━━━━
总字数：2,850 字
章节结构：
├── # 本周概览（300 字）✅
├── ## 热点排行
│   ├── 1. OpenAI 发布（450 字）✅
│   ├── 2. Meta 开源（380 字）✅
│   ├── ...
│   └── 8. 行业融资（200 字）✅
├── ## 趋势分析（500 字）✅
├── ## 下周展望（300 字）✅
└── ## 来源参考（自动生成）

⚠️ 建议：
- 第 3 条"国内厂商动态"可独立成小节
- 趋势分析中"开源 vs 闭源"对比值得展开
```

**通过标准：** 结构逻辑清晰，无重大缺失。用户可提出结构调整。

**回流规则：** 结构重大调整 → 回到 L2 修改 body.md。

---

### 2.2 事实准确性

**确认内容：**

- 关键事实（公司名/产品名/版本号/日期/数字）是否与原始来源一致？
- 是否有交叉验证（每个关键事实至少 2 个独立来源或 1 个官方来源）？
- 是否有未经验证的信息未标注？

**自检清单（Agent 在提交确认前自查）：**

```
□ 所有产品/公司名称拼写正确
□ 所有日期与来源一致
□ 所有数量和数字与来源一致
□ 每个关键论断有对应引用标注 [^ref-xxx]
□ 不确定信息已标"未经独立验证"或"据XX报道"
□ 没有杜撰的引语、数据、发布日期
```

**呈现方式：**

```
✅ 事实核查
━━━━━━━━━━━━━━━━━━━━
已核查事实：12 项
通过：12 项
标注为"未经独立验证"：2 项
  • item-007："据多家媒体报道"（仅有 community 来源）
  • item-015："网友反映"（社交媒体信号，未交叉验证）

交叉验证来源对数：8 对（每对 ≥2 个独立来源）
⚠️ 未交叉验证：4 条（仅有单一来源），已在 editorial_note 标注
```

**通过标准：** 核心事实（标题级/排行级）全部核查通过。标注清晰的信息不阻塞。

**回流规则：** 发现事实错误 → 立即修正 body.md + content_items 并重新提交确认。

---

### 2.3 引用完整性

**确认内容：**

- body.md 中所有 `[^ref-xxx]` 脚注是否在 references[] 中有对应条目？
- references[] 中所有条目是否包含了必需字段（id / title / url / source_type / reliability / access_status）？
- content_items[] 中 status=selected 且有 url 的 item 是否都在 references[] 中有对应记录？
- 图片 `![](assets/xxx)` 是否都在 assets/ 中存在？

**呈现方式：**

```
🔗 引用完整性检查
━━━━━━━━━━━━━━━━━━━━
body.md 脚注：12 个
references[] 条目：12 个
匹配率：12/12 ✅

content_items[selected] 有 url：10 个
对应 references[] 条目：10 个
匹配率：10/10 ✅

图片引用：3 个
assets/ 中对应文件：3 个
匹配率：3/3 ✅

⚠️ 注意：ref-008 链接指向付费墙内容（已标注）
```

**通过标准：** 所有引用匹配 100%。如有不匹配，修正后再提交。

**回流规则：** 引用不匹配 → 回到 L2 修正 manifest.json 或 body.md。

---

### 2.4 图片资源（如有）

**确认内容（L2 生成时如已包含图片）：**

- 图片是否已下载/生成并存放到 assets/ 目录？
- 图片文件名是否规范（`<描述>-<序号>.png` 如 `openai-gpt-5o-01.png`）？
- 图片是否有版权/来源说明（截图来源、生成工具）？

**注：** 图片非 L2 必需项。用户可在内容确认时要求补充图片，或在 L3 格式消费时指定。

## 四、快速通过条件

以下情况下内容确认可简化：

1. **用户明确跳过**："内容确认跳过""直接格式化"。
2. **小修改**：仅修改了个别措辞/排版，未改事实和引用关系。
3. **重复模板型内容**：同样结构、同类信息的前 N 次已验证过流程。

简化时仍需检查引用完整性（自动匹配即可，无需人工逐条确认）。
