## 二、周/月热点分析（hotspot-analysis）— Phase 1

### 2.1 定义

周期性 AI/科技热点复盘。从多个平台聚合过去一周/一月热度最高的新闻和话题，按热度排序，补充趋势分析和编辑视角。

### 2.2 适用场景

- "本周 AI 圈发生了什么"
- "6 月 AI 热点回顾"
- "最近科技圈值得关注的事"
- 固定栏目（每周一/每月 1 日发布）

### 2.3 核心特征

| 维度 | 说明 |
|------|------|
| **驱动模式** | 数据驱动（热点信号 + 趋势分析） |
| **信息获取** | 热点发现 + 趋势信号 + 周期对比 |
| **筛选方式** | 评分制：热度(1-5) + 来源权威(1-5) + 编辑判断(1-5)，总分 ≥ 10 入选 |
| **结构模板** | templates/content-structure/hotspot-analysis.md |
| **时效性** | `weekly` 或 `monthly` |
| **确认关卡** | 信息确认（选题范围、候选数量、来源覆盖）→ 内容确认（结构、事实、引用）→ 格式确认 |

### 2.4 与日报的差异

| | 热点分析 | 日报 |
|--|---------|-----|
| 时间跨度 | 一周/一月 | 一天 |
| 深度 | 趋势分析 + 周期对比 + 预测 | 快速速览 |
| 数量 | 10-20 条，分组呈现 | 5-10 条 |
| 结构 | 概览 → 排行 → 趋势 → 展望 | 简讯式列表 |
| 评分 | 有（热度/权威/编辑判断） | 无（时效性本身就筛选了） |

## 三、GitHub 项目推荐（github-project）— Phase 1

### 3.1 定义

面向 AI/科技从业者和开发者的开源项目推荐。以编辑判断为核心，从 GitHub 发现高质量、实用或前瞻性的开源项目，按适用场景或技术栈分组呈现。

### 3.2 适用场景

- "最近有什么好的 AI 开源项目"
- "推荐几个好用的开发工具库"
- "2026 年值得关注的 GitHub AI 项目"
- 固定栏目或按需选题

### 3.3 核心特征

| 维度 | 说明 |
|------|------|
| **驱动模式** | 编辑判断驱动（主题先行 → 深度检索） |
| **信息获取** | GitHub 发现 + 网页验证 + 社区信号 |
| **筛选方式** | **不评分**，全量编辑判断：stars / 最近提交 / 文档质量 / 社区活跃 / 实用性 |
| **分组逻辑** | 按适用场景（如"模型部署""开发工具""数据可视化"）或技术栈分组 |
| **结构模板** | templates/content-structure/github-project.md |
| **时效性** | `evergreen`（项目推荐不像新闻有时效性） |
| **确认关卡** | 信息确认（项目数量、来源覆盖）→ 内容确认（结构、事实、链接）→ 格式确认 |

### 3.4 筛选标准速查

| 维度 | 关注点 | 不关注 |
|------|-------|--------|
| 项目活跃度 | 最近 3 个月内是否有提交/Release | Stars 绝对数量（可刷） |
| 文档质量 | README 是否清晰、有无 Getting Started | 是否"网红项目" |
| 实用性 | 解决的问题是否明确、是否有真实用户 | 技术是否"炫" |
| 社区 | Issues/Discussions 是否活跃、PR 响应速度 | 是否"大厂出品" |
| 许可证 | 是否为开源许可证（MIT/Apache/GPL 等） | — |

## 四、日报（daily-news）— Phase 2

### 4.1 定义

当日 AI/科技要闻速览。时效性最强的内容类型，标准化生产。从当日热点中筛选 5-10 条最重要新闻，以简讯形式呈现，不做深度分析。每个条目 2-3 句话概括核心事实 + 来源引用。

### 4.2 适用场景

- "今天 AI 圈发生了什么"
- "今日科技要闻速览"
- "帮我整理今天的 AI 新闻"
- 每日固定栏目（每天早上/晚间推送）

### 4.3 核心特征

| 维度 | 说明 |
|------|------|
| **驱动模式** | 数据驱动（热点发现→时效筛选），标准化生产 |
| **信息获取** | 热点发现（≥15 条候选）+ 来源验证 |
| **筛选方式** | **不评分**。时效性本身就是筛选器：`published_at` 必须在 24h 内。超过 24h 的剔除。在时效门槛内按编辑判断（行业重要性、用户兴趣）精选 5-10 条 |
| **结构模板** | templates/content-structure/daily-news.md |
| **时效性** | `daily` 或 `real-time` |
| **确认关卡** | 信息确认（候选数量、时效覆盖）→ 内容确认（结构、事实、引用）→ 格式确认 |
| **典型长度** | 500-1500 字 |
| **输出频率** | 每日 |

### 4.4 与热点分析的差异

| | 日报 | 热点分析 |
|--|-----|---------|
| 时间跨度 | 24h 内 | 7-30 天 |
| 驱动方式 | 时效驱动（时间窗口本身就是筛选器） | 数据驱动（热度+权威+编辑 评分制） |
| 深度 | 简讯为主（2-3 句/条） | 趋势分析 + 周期对比 + 展望 |
| 数量 | 5-10 条 | 10-20 条 |
| 结构 | 速览→要闻列表→明日关注 | 概览→排行→趋势→展望 |
| 评分 | 无 | 有（热度/权威/编辑判断） |
| 趋势分析 | 无 | 有 |
| 周期对比 | 无 | 有 |
| 信号块 | 无（仅编辑视角可选） | 有（💡📈👀⚠️） |

### 4.5 日报特有约束

- ❌ 不需要 `trend_signal`（趋势分析）
- ❌ 不需要 `period_comparison`（周期对比）
- ❌ 不需要评分（时效性本身就是筛选）
- ✅ 每条新闻必须有明确的发布时间（`published_at`），且必须在 24h 内
- ✅ 超过 24h 但有重要延续性的事件（如持续多日的发布会）→ 保留但 editorial_note 注明
- ✅ 每条 2-3 句话即可，不展开分析

## 五、通用专题分析（topic-deep-dive）— Phase 2

### 5.1 定义

公司、技术、事件、产业议题的深度分析。编辑判断驱动的灵活结构，不设固定模板。信息获取依赖多源深度检索（官方+媒体+学术+社区）。分析深度要求多角度覆盖（商业/技术/竞争/影响），按主题覆盖度判断完成度，不设硬性数量门槛。

### 5.2 适用场景

- "深度分析一下 OpenAI 的商业策略"
- "Transformer 架构的技术演进和未来方向"
- "AI 芯片市场的竞争格局"
- "解读欧盟 AI 法案对行业的影响"
- "某公司/某事件的来龙去脉和后续影响"

### 5.3 核心特征

| 维度 | 说明 |
|------|------|
| **驱动模式** | 编辑判断驱动（选题先行→多源深度检索→灵活组织） |
| **信息获取** | 多源深度检索：官方公告 + 主流科技媒体 + 学术论文 + 社区讨论 |
| **筛选方式** | **按主题覆盖度判断**，不评分。判断标准：是否覆盖了该专题的关键角度？信息是否充分到可支撑独立结论？是否有至少 2 个独立来源交叉验证核心事实？ |
| **结构模板** | templates/content-structure/topic-deep-dive.md（灵活变体，非强制） |
| **时效性** | `evergreen` 或 `monthly`（深度分析不追求实时性） |
| **确认关卡** | 信息确认（主题边界、检索覆盖度）→ 内容确认（结构、事实、引用）→ 格式确认 |
| **典型长度** | 2000-6000 字 |
| **输出频率** | 按需 |

### 5.4 结构变体

通用专题不设固定结构，按主题类型灵活选择。以下为推荐结构变体：

| 主题类型 | 推荐结构 |
|---------|---------|
| **公司分析** | 概览→业务拆解→竞争格局→前景判断 |
| **技术分析** | 概览→技术原理→应用场景→局限与展望 |
| **事件分析** | 时间线→影响分析→各方反应→后续走向 |
| **产业分析** | 概览→市场格局→关键玩家→趋势判断→风险提示 |

可选加载 `references/frameworks/` 下的分析框架（PEST/五力/SWOT 等）作为辅助工具，但不强制使用。数据不足时宁可标注"数据局限"也不强行套用框架。

### 5.5 与相邻类型的差异

#### 5.5.1 与 GitHub 项目推荐的关键差异

| | 通用专题 | GitHub 项目 |
|--|---------|------------|
| 主题范围 | 公司/技术/产业/事件/趋势 | 仅开源项目 |
| 信息源 | 多源交叉（官方+媒体+学术+社区） | 以 GitHub 为主 |
| 分析深度 | 多角度（商业/技术/竞争/影响/前景） | 项目评估为主（活跃度/文档/实用） |
| 结构 | 灵活变体，按主题类型切换 | 固定：概览→分组→对比→建议 |
| 时效性 | evergreen（可参考最新也可回溯历史） | evergreen |
| 分组 | 按分析角度（非固定） | 按场景/技术栈（必须分组） |

#### 5.5.2 与热点分析的差异

| | 通用专题 | 热点分析 |
|--|---------|---------|
| 驱动 | 选题先行（用户指定或编辑提出） | 数据驱动（热点信号触发） |
| 时间范围 | 不限定（可跨数月甚至数年） | 固定周期（周/月） |
| 分析深度 | 深度（2000-6000 字） | 中度（1500-4000 字） |
| 趋势分析 | 如果是技术/产业类专题则核心板块 | 可选板块 |
| 评分 | 无 | 有 |

### 5.6 通用专题特有约束

- ✅ 可选加载 `references/frameworks/` 分析框架，但不强制
- ✅ 结构灵活，按主题类型选择变体，不强套模板
- ✅ 确认标准按主题覆盖度判断（非硬性数量门槛）
- ✅ 多源交叉验证：核心事实至少 2 个独立来源
- ❌ 不需要 `trending_discovery`（不走热点流）
- ❌ 不需要评分

## 六、产品/工具推荐（product-recommendation）— Phase 3

### 6.1 定义

面向 AI/科技从业者和用户的**产品/服务推荐**。以编辑判断驱动，从真实使用场景出发，筛选优质 AI 工具、SaaS 服务、模型 API、开发者工具等产品。按用户画像或使用场景分组呈现，侧重功能实用性、性价比和竞品差异化，帮助读者"对号入座"找到适合自己的工具。

**推荐对象明确**：产品/服务/SaaS/模型 API，非开源代码项目（开源代码项目走 github-project）。

### 6.2 适用场景

- "推荐几款好用的 AI 视频生成工具"
- "本地推理工具横向对比"
- "2026年值得关注的 AI 开发者工具"
- "XX 类产品哪个最好用"
- 固定栏目或按需选题

### 6.3 核心特征

| 维度 | 说明 |
|------|------|
| **驱动模式** | 编辑判断驱动（选题先行 → 产品发现 → 评测收集 → 筛选分组） |
| **信息获取** | 官网调研 + 评测检索 + 用户反馈收集 + 竞品对比 |
| **筛选方式** | **不评分**，全量编辑判断：功能完整性 / 价格合理性 / 用户体验 / 竞品差异化 / 适用人群匹配度 |
| **分组逻辑** | 按用户画像（如"新手""专业用户"）或使用场景（如"轻量级内部使用""生产级 API 服务"）分组 |
| **结构模板** | templates/content-structure/product-recommendation.md |
| **时效性** | `evergreen`（产品推荐不像新闻有时效性，但价格/功能可能随时间变化） |
| **确认关卡** | 信息确认（选题范围、产品数量、评测覆盖）→ 内容确认（结构、事实、引用）→ 格式确认 |
| **典型长度** | 1500-3000 字 |
| **输出频率** | 按需 |

### 6.4 筛选标准速查

| 维度 | 关注点 | 不关注 |
|------|-------|--------|
| 功能完整性 | 是否覆盖该品类的核心需求？有无明显功能缺口？ | 功能数量是否"多"（功能堆砌不等于好用） |
| 价格合理性 | 定价是否与功能匹配？有无免费层/试用？ | 是否"最便宜"（便宜但不好用没意义） |
| 用户体验 | 上手难度、文档质量、界面设计、社区支持 | 是否"大厂出品" |
| 竞品差异化 | 与同类产品相比有什么独特优势？ | 是否"最火/最热门" |
| 适用人群匹配 | 是否有明确的适用人群画像？ | 是否"适合所有人"（没有产品适合所有人） |

### 6.5 产品卡片字段

每个推荐产品包含以下信息（比 github-project 多价格/竞品维度，少 stars/语言/许可证）：

| 字段 | 说明 | 必需 |
|------|------|------|
| 名称 | 产品名（可含版本号） | ✅ |
| 一句话定位 | 用一句话说清"这是做什么的" | ✅ |
| 核心功能 | 3-5 个关键功能点 | ✅ |
| 价格 | 免费/付费/分层定价（标注价格来源日期） | ✅ |
| 适用人群 | 谁适合用这个产品 | ✅ |
| 竞品对比 | 同类替代品及差异点 | 推荐（同组 ≥2 个产品时） |
| 编辑点评 | 编辑视角：推荐理由/已知局限/注意事项 | ✅ |

### 6.6 与 GitHub 项目推荐的关键差异

| | 产品推荐 | GitHub 项目 |
|--|---------|------------|
| 推荐对象 | 产品/服务/SaaS/模型 API | 开源代码项目 |
| 信息源 | 官网 + 评测 + 用户反馈 + 竞品对比 | 以 GitHub 页面 + 社区讨论为主 |
| 关键维度 | 功能/价格/体验/竞品对比/适用人群 | 代码质量/文档/社区活跃度/stars/许可证 |
| 卡片字段 | 名称/定位/功能/价格/适用人群/竞品/点评 | 名称/描述/stars/语言/许可证/特性/适用场景/点评 |
| 对比维度 | 功能差异 + 价格 + 适用人群 | 定位 + stars + 性能 + 适用场景 |
| 时效性 | evergreen（产品会更新迭代） | evergreen（代码持续演进） |
| 评分 | 无 | 无 |

**核心边界**：如果推荐对象是"GitHub 上的开源代码项目"→ 走 github-project。如果推荐对象是"可以注册使用、有定价页面、面向终端用户的产品/服务"→ 走 product-recommendation。模型 API 虽在官方文档中有代码示例，但付费 API 本质是产品/服务，走 product-recommendation。

### 6.7 产品推荐特有约束

- ❌ 不需要 `trending_discovery`（不走热点流）
- ❌ 不需要 `trend_signal`（不做趋势分析）
- ❌ 不需要评分（全量编辑判断）
- ✅ 每个产品必须有真实的价格信息（标注来源日期）
- ✅ 同组 ≥2 个产品时推荐加对比表
- ✅ 评测收集：搜索"产品名 + 评测/使用体验"获取真实用户反馈
- ✅ 价格信息标注获取日期，注明"价格可能随时间变化"
- ✅ 不推荐未经实际测试的产品（至少通过评测/用户反馈交叉验证其可用性）

## 七、AI 与科技社会洞察（social-insight）— Phase 3

### 7.1 定义

分析**技术变化对社会、职业、教育、文化、监管的外部影响**。区别于"发生了什么"的热点分析和"技术本身如何"的专题分析——社会洞察关注的是"这意味着什么"和"它如何改变我们的生活"。编辑判断驱动，多源交叉（官方+媒体+学术+社区+数据报告），引入社会学/经济学视角但不做纯学术论文。

**核心判断标准**：一篇文章属于 social-insight 而非 topic-deep-dive 的判定——如果删掉"技术"这个主语后文章仍然成立（如讨论劳动力市场变化、教育体系重构），就是社会洞察；如果离开"技术"文章就散了（如讨论某公司产品矩阵、某技术架构演进），就是通用专题。

### 7.2 适用场景

- "AI 编程助手如何改变初级开发者的职业路径"
- "开源大模型对 AI 创业生态的蝴蝶效应"
- "AI 教育工具的普及会拉大还是缩小教育差距"
- "AI 生成内容对创作者经济的冲击"
- "各国 AI 监管路径的差异与影响"
- "技术性失业：哪些岗位最先被重构"

### 7.3 核心特征

| 维度 | 说明 |
|------|------|
| **驱动模式** | 编辑判断驱动（选题先行 → 多源深度检索 → 多角度交叉分析） |
| **信息获取** | 多源交叉检索：官方报告 + 主流媒体 + 学术论文 + 社区讨论 + 统计数据/行业报告 |
| **筛选方式** | **按角度覆盖度判断**，不评分。判断标准：是否覆盖了至少 2 个分析维度？核心观点是否至少 2 个独立来源支撑？信息是否充分到可支撑独立判断？ |
| **结构模板** | templates/content-structure/social-insight.md（灵活结构，按选题类型选择变体） |
| **时效性** | `evergreen` 或 `monthly`（社会变化缓慢，不追求实时性） |
| **确认关卡** | 信息确认（选题边界、多源覆盖）→ 内容确认（结构、事实、引用、观点与事实分离）→ 格式确认 |
| **典型长度** | 2000-5000 字 |
| **输出频率** | 按需 |

### 7.4 分析维度

社会洞察覆盖 6 个分析维度，每篇文章选择 2-3 个最相关的深度展开，不强制覆盖所有维度：

| 维度 | 示例问题 | 适用条件 |
|------|---------|---------|
| 经济影响 | 技术如何改变产业格局/商业模式/劳动力市场？ | 有可量化数据（成本变化/就业数据/市场规模）时优先 |
| 职业影响 | 哪些岗位被替代/重构/创造？技能需求如何变化？ | 有具体职业案例或行业报告支撑时展开 |
| 教育影响 | 技术如何改变学习方式/教育公平/课程体系？ | 有教育机构实践案例或学术研究时展开 |
| 文化影响 | 技术如何改变创作方式/信息消费/社会认知？ | 主题涉及内容生态/创作工具/信息传播时展开 |
| 伦理影响 | 技术带来哪些公平/隐私/偏见/权力集中问题？ | 所有社会洞察文章均应至少提及伦理维度 |
| 监管影响 | 各国如何应对？监管路径有何差异？影响如何？ | 有明确的政策/法规/监管动态时展开 |

### 7.5 与相邻类型的差异

#### 7.5.1 与通用专题（topic-deep-dive）的关键差异

| | 社会洞察 | 通用专题 |
|--|---------|---------|
| 分析焦点 | 技术的**外部影响**（对社会/职业/教育/文化/伦理/监管） | 技术/公司/产业**本身**（原理/业务/竞争/趋势） |
| 核心问题 | "这意味着什么？如何改变我们的生活？" | "这是什么？为什么重要？如何运作？" |
| 信息源特征 | 必须有学术/社会研究视角 + 宏观数据 | 以官方+媒体+技术分析为主 |
| 分析框架 | 社会经济视角（劳动力市场/教育公平/文化变迁/公共政策） | 商业/技术视角（竞争/架构/商业模式/趋势） |
| 学术引用 | 推荐引用学术研究（标注年份+样本量） | 可选，不强制 |
| 编辑立场 | 不做价值判断（"好/坏"），做影响分析（"可能导致…"） | 可以有编辑观点，标注为"编辑判断" |

**边界判定示例**：
- "Transformer 架构的技术演进" → topic-deep-dive（技术本身）
- "Transformer 驱动的 AI 写作工具如何影响内容行业就业" → social-insight（技术的外部影响）
- "OpenAI 的商业策略与竞争分析" → topic-deep-dive（公司本身）
- "OpenAI 的快速迭代如何改变 AI 创业公司融资逻辑" → social-insight（对产业生态的外部影响）

#### 7.5.2 与热点分析（hotspot-analysis）的关键差异

| | 社会洞察 | 热点分析 |
|--|---------|---------|
| 驱动 | 选题先行（因"值得思考"而写） | 数据驱动（因"热度高"而写） |
| 时间性 | 不追求时效性（社会变化慢） | 必须周期性（周/月） |
| 深度 | 深度（2000-5000 字，多角度交叉） | 中度（1500-4000 字，排行+趋势） |
| 评分 | 无 | 有（热度/权威/编辑判断） |
| 叙事线 | 现象→分析→深层影响→讨论与展望 | 概览→排行→趋势→展望 |

### 7.6 社会洞察特有约束

- ❌ 不需要 `trending_discovery`（不走热点流）
- ❌ 不需要 `trend_signal`（不做趋势信号分析）
- ❌ 不需要评分
- ✅ 核心观点至少 **2 个独立来源**支撑
- ✅ 引用学术研究时标注**论文年份**和**样本量**（如"2025 年，n=1,200"）
- ✅ 不确定的影响标注**"需更长期数据验证"**
- ✅ **编辑观点与事实明确分开**：事实用陈述句+脚注引用，编辑观点用"编辑判断"标注
- ✅ **不做价值判断**（"AI 会摧毁创意行业"），**做影响分析**（"AI 工具使个人创作者产能提升 3-5 倍，可能导致入门级内容岗位需求下降"）
- ✅ 可选加载 `references/frameworks/` 下的社会经济学分析框架，但不强制
- ✅ 多源交叉：官方 + 媒体 + 学术 + 社区 + 数据报告，至少覆盖 3 类来源

## 八、推荐内容建议（content-advice）— Phase 2

### 8.1 定义

选题推荐、标题方向、传播角度建议。轻量信息扫描 + 编辑判断。对当日/近期热点进行快速扫描，从中发现有内容价值的选题，输出结构化建议文档。**不产出完整 content-package**，仅产出建议卡片。

### 8.2 适用场景

- "最近有什么值得写的选题"
- "这个方向可以怎么写？帮我出几个标题方向"
- "今天有什么热点可以追"
- 编辑晨会/选题会前的素材准备

### 8.3 核心特征

| 维度 | 说明 |
|------|------|
| **驱动模式** | 信号扫描驱动（轻量，非完整采编） |
| **信息获取** | 单次热点扫描（≥10 条候选信号） |
| **筛选方式** | 编辑快速判断（选题潜力、传播性、与目标读者匹配度），不评分 |
| **输出** | **建议文档**（非 content-package），不进入 L3 格式消费 |
| **结构模板** | templates/content-structure/content-advice.md |
| **时效性** | 不适用（建议文档本身不发布，是内部选题素材） |
| **确认关卡** | 仅 1 层（选题确认），可选 |
| **典型长度** | 300-1000 字（建议卡片格式） |
| **输出频率** | 按需（每日晨会/选题会前） |

### 8.4 与相邻类型的差异

#### 8.4.1 与日报的差异

| | 内容建议 | 日报 |
|--|---------|-----|
| 输出物 | 建议文档（选题池） | content-package（完整内容资产） |
| 确认流程 | 1 层（可选） | 3 层（信息→内容→格式） |
| 后续流转 | 用户挑选后→进入对应 workflow 生产 | 完成后直接进入 L3 格式消费 |
| 传播 | 内部使用，不发布 | 对外发布 |
| 结构 | 建议卡片式（选题+理由+标题方向） | 完整文章结构 |

#### 8.4.2 与通用专题的差异

| | 内容建议 | 通用专题 |
|--|---------|---------|
| 深度 | 轻量扫描（信号级别） | 深度分析（2000-6000 字） |
| 检索深度 | 单次扫描即可 | 多源深度检索 |
| 输出 | 选题池（多个建议） | 单篇深度分析 |

### 8.5 建议卡片格式

每个建议卡片包含以下字段（非强制，可根据扫描结果调整）：

```
- 选题概述：一句话描述选题内容
- 信号来源：哪些热点/趋势/事件支撑这个选题（2-3 条信号）
- 推荐理由：为什么值得写（时效性/用户兴趣/行业重要性）
- 标题方向：2-3 个可能的标题（标注风格：资讯型/分析型/观点型）
- 传播角度：适合什么平台、什么读者群、什么时机发布
- 建议内容类型：下一步进入哪个 workflow（daily-news / topic-deep-dive / hotspot-analysis 等）
```

### 8.6 内容建议特有约束

- ❌ **不产出 content-package**（不生成 manifest.json + body.md）
- ❌ 不需要完整来源验证（信号级别不要求三项硬检查）
- ❌ 不进入 L3 格式消费
- ✅ 输出为 Markdown 建议文档，可保存到 `outputs/content-advice-<date>.md`
- ✅ 确认仅 1 层（选题确认），用户可快速过一遍后挑选
- ✅ 用户挑选的选题 → 进入对应 workflow 从 L1 开始完整生产
