## 一、核心原则：结构灵活，按主题选型

通用专题不设固定模板。Agent 根据主题类型选择最合适的结构变体，并可混合使用多个变体的元素。

**选择流程：**

```
用户需求 / 编辑选题
    │
    ▼
判断主题类型（公司/技术/事件/产业）
    │
    ▼
选择对应结构变体（见第二节）
    │
    ▼
按实际情况灵活调整（可跨变体借用元素）
```

## 二、四种结构变体

### 2.1 公司分析 — 结构

```
# <标题>
## 一、公司概览
## 二、业务拆解
## 三、竞争格局
## 四、优势与风险
## 五、前景判断
## 参考来源
```

#### 2.1.1 一、公司概览（200-400 字）

**内容块：**

- 公司基本信息（成立时间/总部/规模/融资情况）
- 核心业务定位（一句话说清"这家公司做什么"）
- 当前阶段（初创/成长/成熟/转型）
- 分析焦点（本次分析重点关注哪个方面）

#### 2.1.2 二、业务拆解（600-1500 字）

**内容块：**

- 主营业务线及其收入/用户规模
- 核心技术/产品能力
- 商业模式（to B / to C / API / 开源+服务 等）
- 近期关键动态（产品发布/合作/人事变动）

#### 2.1.3 三、竞争格局（600-1200 字）

**内容块：**

- 主要竞争对手及各自定位
- 差异化优势 vs 竞争对手
- 市场份额/竞争态势
- 可选：对比表格

#### 2.1.4 四、优势与风险（300-600 字）

**内容块：**

- 核心护城河（技术/数据/生态/品牌/人才）
- 主要风险（竞争/技术/监管/市场）
- 争议/负面信息（如实呈现，不回避）

#### 2.1.5 五、前景判断（200-400 字）

基于以上分析的编辑判断（非预测），说明理由和不确定性。

**约束：**
- ✅ 基于事实和逻辑推理，标注推理依据
- ❌ 不做"XX 一定成功/失败"的断言

### 2.2 技术分析 — 结构

```
# <标题>
## 一、技术概览
## 二、技术原理（深入浅出）
## 三、应用场景
## 四、局限与挑战
## 五、发展展望
## 参考来源
```

#### 2.2.1 一、技术概览（200-400 字）

- 这项技术是什么（一句话概括）
- 解决什么问题
- 当前阶段（研究/早期/成熟/衰退）
- 为什么现在值得关注

#### 2.2.2 二、技术原理（600-1500 字）

- 核心原理（深入浅出，避免纯学术语言）
- 关键技术组件/架构
- 与同类/前代技术的差异
- 可用图示/类比降低理解门槛

#### 2.2.3 三、应用场景（400-800 字）

- 已落地的实际应用
- 可预见的潜在应用
- 典型用户/客户画像
- 各场景的成熟度

#### 2.2.4 四、局限与挑战（300-600 字）

- 技术局限性（性能/成本/适用条件）
- 落地障碍（基础设施/人才/监管）
- 已知问题/争议

#### 2.2.5 五、发展展望（200-400 字）

- 短期（6-12 个月）可预期的进展
- 长期（2-5 年）可能走向
- 关键变量（什么因素会改变预期轨迹）

### 2.3 事件分析 — 结构

```
# <标题>
## 一、事件概述
## 二、时间线
## 三、影响分析
## 四、各方反应
## 五、后续走向
## 参考来源
```

#### 2.3.1 一、事件概述（200-300 字）

- 发生了什么（一句话 + 关键数字）
- 事件性质（发布/收购/事故/政策/人事/纠纷）
- 为什么这个事件重要

#### 2.3.2 二、时间线（300-800 字）

按时间顺序呈现事件关键节点。格式：

```markdown
| 时间 | 事件 |
|------|------|
| 6月10日 | XXX 首次传出消息 |
| 6月12日 | 官方确认并发布公告 |
| 6月15日 | 竞品 YYY 作出回应 |
```

**原则：** 只列可确认时间的事实。不确定时间的标注"约""据传"。

#### 2.3.3 三、影响分析（600-1200 字）

多角度评估事件影响：

- 对行业格局的影响
- 对用户/消费者的影响
- 对相关方（供应链/合作伙伴/投资者）的影响
- 短期 vs 长期影响

#### 2.3.4 四、各方反应（300-600 字）

- 直接相关方表态
- 竞争对手反应
- 行业分析师/媒体评论
- 社区/用户反应

**呈现方式：** 可按"支持方 / 反对方 / 中立方"分组，或按"官方 / 行业 / 社区"分层。

#### 2.3.5 五、后续走向（200-400 字）

- 已公布的后续步骤
- 可预见的连锁反应
- 需要持续关注的关键节点

### 2.4 产业分析 — 结构

```
# <标题>
## 一、产业概览
## 二、市场格局
## 三、关键玩家
## 四、驱动因素与趋势
## 五、风险与不确定性
## 参考来源
```

#### 2.4.1 一、产业概览（200-400 字）

- 产业定义和边界
- 市场规模（数字 + 增长率 + 来源）
- 发展阶段（新兴/成长/成熟/转型）
- 分析视角（本次关注什么维度）

#### 2.4.2 二、市场格局（400-800 字）

- 产业链图谱（上中下游）
- 市场集中度（头部玩家份额）
- 区域分布（中美欧日韩等）
- 可选：产业链图示（结构化的 Markdown 表格）

#### 2.4.3 三、关键玩家（600-1200 字）

按类别分组呈现主要参与者：

| # | 类别 | 代表企业/机构 | 核心能力 |
|---|------|-------------|---------|
| 1 | 头部厂商 | XXX | ... |
| 2 | 挑战者 | YYY | ... |
| 3 | 新进入者 | ZZZ | ... |

每组展开：定位、优势、策略、近期动态。

#### 2.4.4 四、驱动因素与趋势（400-800 字）

- 技术驱动因素
- 市场需求驱动因素
- 政策/监管驱动因素
- 1-3 个核心趋势判断

#### 2.4.5 五、风险与不确定性（200-400 字）

- 监管风险
- 技术路线不确定性
- 市场竞争风险
- 经济周期影响

## 三、可选：加载分析框架

`references/frameworks/` 目录下的分析框架可作为辅助工具按需加载。

### 3.1 可用框架（按适用场景）

| 框架 | 适用场景 | 文件 |
|------|---------|------|
| PEST 分析 | 产业/宏观环境分析 | `references/frameworks/pest.md` |
| 波特五力 | 行业竞争结构分析 | `references/frameworks/porter-five-forces.md` |
| SWOT 分析 | 公司/产品优劣势分析 | `references/frameworks/swot.md` |
| 技术成熟度曲线 | 技术发展阶段判断 | `references/frameworks/hype-cycle.md` |

### 3.2 使用规则

- **可选、不强制**：框架是辅助工具，不替代编辑判断。
- **数据不足时不用**：宁可标注"数据局限"也不强行套框架填内容。
- **不改变结构**：框架的结论融入已有结构变体，不额外增加"SWOT 分析"等独立章节。

## 四、通用约束（所有变体适用）

### 4.1 内容约束

- ✅ 纯 Markdown（标题、段落、列表、表格、引用、链接、脚注）
- ✅ 关键事实用 `[^ref-xxx]` 脚注标注来源
- ✅ 所有数据标注来源（年份、统计口径）
- ❌ 不含 HTML/CSS
- ❌ 不含评分信息
- ❌ 不含格式指令
- ❌ 不做无来源的预测（"可能""预计"等措辞需有推理依据）

### 4.2 叙事质量

- **开头有力**：概览章节必须让读者 30 秒内理解"这篇分析在讲什么、为什么重要"
- **逻辑链清晰**：每个分析结论有事实支撑（"因为 X 和 Y，所以 Z"）
- **多角度覆盖**：不只讲"好的一面"或"坏的一面"
- **编辑立场**：可以有编辑观点，但必须标注为"编辑判断"并与事实分开

### 4.3 确认标准（按覆盖度判断，非数量门槛）

专题是否完成，按以下维度判断：

1. **核心事实覆盖**：该专题的核心事件/数据是否完整？
2. **多角度覆盖**：是否从至少 2 个角度（商业/技术/竞争/影响）进行了分析？
3. **来源交叉验证**：核心事实是否至少有 2 个独立来源？
4. **逻辑自洽**：分析结论是否有事实支撑，不自相矛盾？
5. **局限声明**：是否诚实地标注了数据局限和不确定性？
