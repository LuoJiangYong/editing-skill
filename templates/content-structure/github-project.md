## 二、各章节详解

### 2.1 标题

**格式：** `<主题/场景> 开源项目推荐` 或 `推荐 N 个 <描述> 的 GitHub 项目`

**示例：**
- `2026年6月 AI 开源项目推荐`
- `推荐 5 个提升 AI 应用部署效率的开源工具`
- `本地运行大模型？这 8 个开源项目值得关注`

**原则：** 标题应体现推荐主题或适用场景，让读者一眼知道"这些项目对我有什么用"。

---

### 2.2 一、推荐概览（200-400 字）

**目的：** 说明本期推荐的背景和选题理由。

**内容块：**

1. **选题理由**：为什么推荐这批项目？（行业趋势/用户需求/技术热点）
2. **筛选说明**：简要说明筛选标准（活跃度/文档/实用性）。
3. **分组预览**：一句话预告本期包含哪几个分组。

**示例：**

```markdown
## 一、推荐概览

随着大模型部署需求激增，本地推理和模型服务化成为开发者关注的焦点。
本期聚焦"模型部署与推理优化"方向，从 GitHub 筛选了 8 个活跃维护、
文档齐全、社区健康的开源项目，按适用场景分为三组。

本期推荐分组：
- 🚀 轻量级本地推理（3 个项目）
- ⚡ 高性能服务化部署（3 个项目）
- 🔧 工具链与辅助库（2 个项目）
```

---

### 2.3 二、项目分组（核心章节，1200-3000 字）

**目的：** 按适用场景或技术栈分组呈现项目，每组 2-5 个项目。

#### 分组标题

按适用场景命名，体现"这个分组的项目解决什么问题"：

```
### 🚀 轻量级本地推理
### ⚡ 高性能服务化部署
### 🔧 工具链与辅助库
### 📊 模型评估与监控
### 🎯 特定领域应用
```

#### 项目卡片（每个项目 150-300 字）

```markdown
#### <项目名>

**一句话描述**：<项目简要介绍，1 句话>

| 属性 | 值 |
|------|-----|
| ⭐ Stars | 12.3k |
| 📝 语言 | Python |
| 📄 许可证 | MIT |
| 🔄 最近更新 | 2026-06-10 |
| 🔗 仓库 | [github.com/xxx/xxx](https://github.com/xxx/xxx) |

**核心特性：**
- <特性 1>
- <特性 2>
- <特性 3>

**适用场景：** <什么情况下应该用这个项目>

**编辑点评：** <编辑视角：为什么推荐/与同类项目的差异/注意事项> [^ref-xxx]
```

**项目卡片约束：**
- Stars 标注精确数字（如 `12.3k`），不写"一万多"。
- 最近更新日期必须真实（从 GitHub 页面获取）。
- 许可证必须准确（MIT / Apache-2.0 / GPL-3.0 / 其他）。
- 适用场景一句话说清，不堆砌关键词。

---

### 2.4 三、对比总览（可选，200-400 字）

**目的：** 当同一分组内项目功能相近时，用对比表帮助读者快速选择。

**呈现方式：**

```markdown
## 三、对比总览

| 项目 | 定位 | Stars | 推理速度 | 模型支持 | 适用场景 |
|------|------|-------|---------|---------|---------|
| llama.cpp | 通用本地推理 | 78k | ⚡⚡⚡ | GGUF | 个人开发者 |
| Ollama | 一键部署体验 | 105k | ⚡⚡ | GGUF | 快速体验 |
| vLLM | 高性能服务 | 45k | ⚡⚡⚡ | 多格式 | 生产环境 |
```

**使用条件：** 分组内 ≥ 3 个项目且功能有可比性时使用。不强行对比性质不同的项目。

---

### 2.5 四、选择建议（200-400 字）

**目的：** 帮助不同需求的读者快速找到适合自己的项目。

**呈现方式：**

```markdown
## 四、选择建议

- **想快速体验本地大模型？** → Ollama，一行命令搞定。
- **需要生产级推理服务？** → vLLM，吞吐量最高。
- **想在边缘设备上跑模型？** → llama.cpp，资源占用最小。
- **需要完整 MLOps 流程？** → 结合 MLflow + BentoML。
```

**原则：** 建议基于项目实际特性，不主观偏好。如果某个项目有明显缺陷（如文档差、长期未维护），在建议中诚实说明。

**格式：** 自动从 manifest.json 的 references[] 生成，包含每个项目的 GitHub 链接和引用的社区评价来源。


## 三、分组逻辑指南

### 3.1 分组维度选择

| 维度 | 适用场景 | 示例 |
|------|---------|------|
| 适用场景 | 项目功能差异大 | 本地推理 / 服务部署 / 工具链 |
| 技术栈 | 跨语言/框架推荐 | Python / TypeScript / Rust |
| 模型规模 | 按算力需求分组 | 消费级 GPU / 服务器集群 / 边缘设备 |
| 用户画像 | 按使用者分组 | 独立开发者 / 小型团队 / 企业级 |

**选择原则：** 选最能帮助读者"对号入座"的维度。优先适用场景，其次技术栈。

### 3.2 分组数量

- 最少：1 组（如主题非常聚焦）
- 推荐：2-4 组
- 最多：5 组
- 每组：2-5 个项目

## 四、项目卡片的可选扩展

以下为可选字段，按需使用：

| 扩展字段 | 用途 | 使用条件 |
|---------|------|---------|
| 对比/替代 | "如果你想找 XX 的替代品" | 项目是某个知名工具的替代 |
| 上手难度 | 🟢 简单 / 🟡 中等 / 🔴 复杂 | 项目间上手难度差异大 |
| 社区活跃度 | Issues 响应速度、PR 合并频率 | 社区健康状况值得特别说明 |
| 已知局限 | 文档、性能、兼容性等问题 | 编辑在调研中发现明显短板 |
